## 1. Executive Summary
At 02:00 AM, an automated certificate rotation script deployed an incorrectly configured certificate to the API Gateway. This triggered a cascade of failures: immediate TLS errors, runaway auto-scaling due to health check failures, Kubernetes resource exhaustion, monitoring blackout, and finally, a database read-only lock. The service remained down for 2.5 hours, significantly exceeding the monthly SLA allowance. The root cause was a lack of validation in the certificate rotation pipeline and a lack of "circuit breakers" in the auto-scaling logic.

---

## 2. The Cascade Map
The incident did not stem from a single point of failure but rather a chain reaction where the failure of one system exacerbated the next.

| Stage | Time | Event | Impact |
| :--- | :--- | :--- | : |
| **1. Initial Trigger** | 02:00 AM | Automated cert rotation deploys `*.example.com` instead of `*.api.example.com`. | **TLS Handshake Failure.** All incoming API requests fail immediately. |
| **2. Secondary Failure** | 02:05 AM | Health checks fail. Auto-scaler interprets this as "under-provisioning" and spins up new pods. | **Resource Inflation.** New pods inherit the bad cert and fail health checks immediately. |

## 3. Root Cause Analysis (RCA)

### Primary Root Cause
**Defective Automated Deployment Pipeline:** The certificate rotation script lacked a validation step to verify the Subject Alternative Name (SAN) matches the expected domain pattern (`*.api.example.com`) before applying the certificate to the production gateway.

### Contributing Factors
1.  **Aggressive Auto-Scaling Logic:** The auto-scaler was configured to scale up on *any* health check failure without a "cooldown" period or a check for systemic errors (like TLS failures vs. application errors).
2.  **Single Point of Failure for Monitoring:** The monitoring stack was hosted on the same Kubernetes cluster as the application. When the cluster control plane died, the monitoring system died with it, creating a "silent failure" scenario.
3.  **Database Connection Pooling:** The application pods did not implement exponential backoff or connection pooling limits, causing a "thundering herd" effect that overwhelmed the database immediately upon scaling.

---

## 4. The 5 Whys

1.  **Why did the service go down?**
    *   Because the API Gateway returned TLS errors, causing all requests to fail.
2.  **Why did the API Gateway return TLS errors?**
    *   Because the automated rotation script deployed a certificate with the wrong domain (`*.example.com`).
3.  **Why did the script deploy the wrong certificate?**
    *   Because the deployment pipeline did not validate the certificate's Subject Alternative Names (SANs) against the expected domain pattern.
4.  **Why did the outage last 2.5 hours instead of minutes?**
    *   Because the auto-scaler created more pods (making the problem worse), the monitoring system went down (hiding the issue), and the database locked (preventing recovery).
5.  **Why was the on-call unaware until a customer complained?**
    *   Because the monitoring system was hosted on the same failing cluster, resulting in a total blackout of alerting capabilities.

## 6. Action Items (Blameless)

- [ ] **Add Certificate Validation:** Implement a pre-flight check in the CI/CD pipeline to verify certificate SANs match the target domain before deployment. (Owner: Platform Team, Due: 48 hours)
