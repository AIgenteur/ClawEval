## PROMPT:
Write tests for this pagination function. It has a SUBTLE BUG — your tests must catch it. 

```python
def paginate(items, page, per_page=10):
    """Return a page of items with metadata."""
    total = len(items)
    total_pages = total // per_page + (1 if total % per_page else 0)
    
    start = (page - 1) * per_page
    end = start + per_page
    
    return {
        "items": items[start:end],
        "page": page,
        "per_page": per_page,
        "total": total,
        "total_pages": total_pages,
        "has_next": page < total_pages,
        "has_prev": page > 1
    }
```

1. Test normal pagination (100 items, page 1, per_page 10)
2. Test last page with partial results
3. Test empty list
4. Test page beyond range
5. Test per_page larger than total items
6. Test exactly divisible (e.g., 20 items, per_page 10)
7. FIND THE BUG (hint: what happens with page=0? or negative page?)

### ✅ **Understanding the Bug**

The `paginate` function does **not** validate the `page` parameter. As a result, if a user passes `page=0` or a negative number, the function will still execute and return data based on that invalid input. This can lead to:

- Incorrect slicing of the `items` list (e.g., returning an empty list when the user expects the last page).
- Incorrect metadata, such as `has_next` being `True` when the page is invalid.

For example, with `page=0`, the start index becomes negative, and the slice `items[start:end]` may return an empty list. However, the `has_next` flag is still computed as `page < total_pages`, which is `True` in this case, even though the page is invalid.
